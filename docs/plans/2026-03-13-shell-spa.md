# Shell SPA — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the SolidJS shell SPA at `web/shell/` — the Module Federation host that renders platform chrome and dynamically mounts service UIs at runtime.

**Architecture:** Thin SolidJS app in the existing `web/` pnpm workspace. Fetches `/api/services` to discover available services, registers Module Federation remotes at runtime via `@module-federation/runtime`, and mounts them into a shell layout built with `@workfort/ui` web components. Auth and theme context shared to remotes via MF shared singletons.

**Tech Stack:** SolidJS 1.9, @solidjs/router, @module-federation/runtime, @module-federation/vite, @workfort/ui, @workfort/ui-solid, @workfort/auth, Vite 6, TypeScript 5

**Spec:** `docs/2026-03-13-shell-spa-design.md`

---

## Deviations from Spec

- Workspace `pnpm-workspace.yaml` updated to include `shell` alongside `packages/*`
- Build pipeline copies SPA output into `cmd/web/placeholder/` (replacing the fallback HTML) instead of a separate `cmd/web/dist/` directory. This avoids changing the embed directive or `fs.Sub` path — the Go embed always reads from `placeholder/`, which contains either the fallback HTML (fresh clone) or the full SPA (after build).

### Task 5: Module Federation remote loader

**Files:**
- Create: `web/shell/src/lib/remotes.ts`

**Context:** Wraps `@module-federation/runtime` to register and load remotes dynamically. `initRemotes()` is called once after fetching `/api/services` — it registers a remote for each service with `ui: true`. `loadServiceModule()` loads a specific remote's exports on demand (lazy, when the user navigates to that service's route).

Remote entry URL pattern: `/api/{service}/ui/remoteEntry.js` — this is proxied by the Go backend to the service's `/ui/remoteEntry.js`.

- [ ] **Step 1: Create remotes.ts**

```typescript
import { init, registerRemotes, loadRemote } from '@module-federation/runtime';
import type { ServiceInfo } from './api';

let initialized = false;

// Bootstrap the MF runtime once (required before registerRemotes).
init({ name: 'shell', remotes: [] });

export function initRemotes(services: ServiceInfo[]): void {
  if (initialized) return;

  const remotes = services
    .filter((s) => s.enabled && s.ui)
    .map((s) => ({
      name: s.name,
      entry: `/api/${s.name}/ui/remoteEntry.js`,
    }));

  if (remotes.length > 0) {
    registerRemotes(remotes);
  }

  initialized = true;
}

export interface ServiceModule {
  default: () => any;
  manifest: { name: string; label: string; route: string; minWidth?: number };
  SidebarContent?: () => any;
  HeaderActions?: () => any;
}

export async function loadServiceModule(
  serviceName: string,
): Promise<ServiceModule> {
  const mod = await loadRemote<ServiceModule>(`${serviceName}/index`);
  if (!mod || !mod.default || !mod.manifest) {
    throw new Error(
      `Remote "${serviceName}" did not export required fields (default, manifest)`,
    );
  }
  return mod;
}
```

### Task 7: Theme store

**Files:**
- Create: `web/shell/src/stores/theme.ts`

**Context:** Manages the `data-theme` attribute on `<html>` and persists to `localStorage`. The `useTheme()` hook from `@workfort/ui-solid` reads the attribute reactively via MutationObserver — this store only owns writes. The toggle function flips between `"dark"` and `"light"`.

Note: Light mode tokens don't exist in `@workfort/ui/style.css` yet. The toggle will work (attribute changes, localStorage persists) but visually nothing changes until light tokens are added. This is expected.

- [ ] **Step 1: Create theme.ts**

```typescript
type Theme = 'dark' | 'light';

const STORAGE_KEY = 'wf-theme';

function getInitialTheme(): Theme {
  if (typeof window === 'undefined') return 'dark';
  return (localStorage.getItem(STORAGE_KEY) as Theme) ?? 'dark';
}

export function applyTheme(theme: Theme): void {
  document.documentElement.setAttribute('data-theme', theme);
  localStorage.setItem(STORAGE_KEY, theme);
}

export function toggleTheme(): void {
  const current = (document.documentElement.getAttribute('data-theme') ?? 'dark') as Theme;
  applyTheme(current === 'dark' ? 'light' : 'dark');
}

// Apply saved theme on load (overrides the static "dark" in index.html if user chose light).
applyTheme(getInitialTheme());
```

### Task 10: Service mount component

**Files:**
- Create: `web/shell/src/components/service-mount.tsx`

**Context:** The core of the shell. Takes a service name, lazily loads the Module Federation remote via `loadServiceModule()`, and renders the remote's default export. Wrapped in `<Suspense>` (shows a skeleton while loading) and `<ErrorBoundary>` (shows `<Unavailable>` on failure). Also exposes the remote's optional `SidebarContent` and `HeaderActions` exports via context or props (passed to the shell layout).

- [ ] **Step 1: Create service-mount.tsx**

```tsx
import { createResource, Suspense, ErrorBoundary, type Component } from 'solid-js';
import { Dynamic } from 'solid-js/web';
import { loadServiceModule, type ServiceModule } from '../lib/remotes';
import Unavailable from './unavailable';

const ServiceMount: Component<{
  name: string;
  label: string;
  onModule?: (mod: ServiceModule | null) => void;
}> = (props) => {
  const [mod] = createResource(
    () => props.name,
    async (name) => {
      const m = await loadServiceModule(name);
      props.onModule?.(m);
      return m;
    },
  );

  return (
    <ErrorBoundary fallback={<Unavailable label={props.label} />}>
      <Suspense fallback={<wf-skeleton width="100%" height="200px" />}>
        {mod() && <Dynamic component={mod()!.default} />}
      </Suspense>
    </ErrorBoundary>
  );
};

export default ServiceMount;
```

### Task 11: Nav bar component

**Files:**
- Create: `web/shell/src/components/nav-bar.tsx`

**Context:** Horizontal bar at the top of the shell. Three sections: WorkFort branding (left), service tabs (center), theme toggle (right). Tabs are built from the services store — only services with `enabled: true` get a tab. Active tab is highlighted based on the current route using `@solidjs/router`'s `useLocation()`. - [ ] **Step 1: Create nav-bar.tsx**

Uses `<wf-list>` and `<wf-list-item>` from `@workfort/ui` for tabs, as required by the spec. Services with `enabled: true` get a clickable tab. Services with `ui: false` get a grayed-out tab (the `active` attribute on `<wf-list-item>` controls the highlight). Clicking a tab navigates via `@solidjs/router`'s `useNavigate()`.

```tsx
import { For, type Component } from 'solid-js';
import { useNavigate, useLocation } from '@solidjs/router';
import { services } from '../stores/services';
import { toggleTheme } from '../stores/theme';
import { useTheme } from '@workfort/ui-solid';

const NavBar: Component = () => {
  const navigate = useNavigate();
  const location = useLocation();
  const theme = useTheme();

  return (
    <nav class="shell-nav">
      <span class="shell-nav__brand">WorkFort</span>
      <wf-list class="shell-nav__tabs">
        <For each={services().filter((s) => s.enabled)}>
          {(svc) => (
            <wf-list-item
              active={location.pathname.startsWith(svc.route)}
              class={!svc.ui ? 'shell-nav__tab--disabled' : ''}
              on:wf-select={() => navigate(svc.route)}
            >
              {svc.label}
            </wf-list-item>
          )}
        </For>
      </wf-list>
      <div class="shell-nav__spacer" />
      <wf-button variant="text" on:wf-click={() => toggleTheme()}>
        {theme() === 'dark' ? 'Light' : 'Dark'}
      </wf-button>
    </nav>
  );
};

export default NavBar;
```

**Context:** The root component. Uses `@solidjs/router` with a `root` layout and a single `/:service/*rest` catch-all route. The route component looks up the service from the services store and renders `ServiceMount` (if `ui: true`) or `Unavailable` (if `ui: false`). The `/` route redirects to the first enabled service. This approach avoids dynamically generating `<Route>` elements (which must be static children of `<Router>` in `@solidjs/router`).

- [ ] **Step 1: Update app.tsx**

Replace the placeholder content:

```tsx
import { type Component, createSignal, createMemo } from 'solid-js';
import { Router, Route, Navigate, useParams } from '@solidjs/router';
import { services } from './stores/services';
import './stores/theme';
import ShellLayout from './components/shell-layout';
import ServiceMount from './components/service-mount';
import Unavailable from './components/unavailable';
import type { ServiceModule } from './lib/remotes';

const App: Component = () => {
  const [sidebarComponent, setSidebarComponent] = createSignal<(() => any) | undefined>();

  const handleModule = (mod: ServiceModule | null) => {
    setSidebarComponent(mod?.SidebarContent ? () => mod.SidebarContent! : undefined);
  };

  const firstRoute = createMemo(() => {
    const enabled = services().find((s) => s.enabled);
    return enabled?.route ?? '/';
  });

  const ServicePage: Component = () => {
    const params = useParams<{ service: string }>();
    const svc = createMemo(() =>
      services().find((s) => s.enabled && s.route === `/${params.service}`),
    );

    return (
      <>
        {svc() ? (
          svc()!.ui ? (
            <ServiceMount name={svc()!.name} label={svc()!.label} onModule={handleModule} />
          ) : (
            <Unavailable label={svc()!.label} />
          )
        ) : (
          <Navigate href="/" />
        )}
      </>
    );
  };

  return (
    <Router root={(props) => <ShellLayout sidebar={sidebarComponent()}>{props.children}</ShellLayout>}>
      <Route path="/:service/*rest" component={ServicePage} />
      <Route path="/" component={() => <Navigate href={firstRoute()} />} />
    </Router>
  );
};

export default App;
```

# SPDX-License-Identifier: Apache-2.0
#MISE description="Build the workfort binary"
#MISE sources=["**/*.go", "go.mod", "go.sum"]
#MISE outputs=["build/workfort"]
set -euo pipefail

# Build shell SPA if web/shell exists and has node_modules
if [ -d "web/shell/node_modules" ]; then
  echo "Building shell SPA..."
  (cd web/shell && pnpm build)
  rm -rf cmd/web/placeholder

mkdir -p build
VERSION=$(git rev-parse --short HEAD 2>/dev/null || echo "dev")
go build -ldflags "-X github.com/Work-Fort/Scope/cmd.Version=${VERSION}" -o build/workfort .
echo "✓ Build complete"
